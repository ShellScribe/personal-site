# Project Brief

## Purpose

This repository hosts a Jekyll GitHub Pages personal portfolio for a senior cloud/platform engineer. The site should present professional credibility, technical judgment, and delivery maturity to engineering leaders, hiring teams, technical peers, and collaborators.

## Target Audience

- Engineering managers and directors evaluating senior platform/cloud experience.
- Staff and senior engineers looking for technical signal and collaboration style.
- Recruiters who need a clear summary of domain strengths.
- Professional contacts looking for writing, projects, talks, or contact paths.

## Desired Positioning

The portfolio should communicate:

- Deep cloud/platform engineering capability.
- Practical experience operating reliable systems.
- Security-conscious engineering habits.
- Clear communication with both technical and business audiences.
- Senior judgment: tradeoffs, maintainability, incident awareness, cost awareness, and pragmatic delivery.

## Content Priorities

Prioritize durable, high-signal content:

- A concise professional summary.
- Selected resume-safe projects or case studies with problem, constraints, actions, and outcomes.
- Cloud/platform capabilities, such as infrastructure as code, CI/CD, observability, reliability, security, networking, automation, developer experience, and containers/orchestration where supported by source material.
- Writing or notes that show technical reasoning.
- Contact and professional profile links.

## Site Constraints

- This is a public static site. Do not include private employer, customer, infrastructure, incident, contract, or security-sensitive details.
- The current implementation uses Jekyll with the `minima` theme. Preserve that foundation unless a theme migration or custom in-repo theme is explicitly requested.
- GitHub Pages compatibility matters. Avoid tools, plugins, or build steps that make deployment fragile.
- This is a GitHub Pages project site, so internal links and assets must work with the configured `baseurl`.
- Keep pages fast, accessible, and readable on mobile.

## Tone

The voice should be confident, specific, and measured. Prefer concrete evidence over grand claims. The site should feel like it belongs to a senior engineer who has shipped, operated, debugged, secured, and explained production systems.

## Non-Goals

- Do not turn the site into a marketing splash page unless explicitly requested.
- Do not add complex front-end application behavior unless it directly supports the portfolio.
- Do not add analytics, trackers, comments, forms, or newsletter tooling without security and privacy review.
- Do not invent experience, certifications, metrics, employers, technologies, or project outcomes.
- Do not include AI scaffold text, prompt notes, internal planning notes, or placeholder process language on public-facing pages.
