# Content Rules

These rules apply to public-facing portfolio text, metadata, links, images, downloadable assets, and project descriptions.

## Accuracy

- Do not invent facts, employers, titles, dates, certifications, metrics, talks, publications, or project outcomes.
- Do not invent technologies, platforms, tools, frameworks, or scale claims.
- Preserve uncertainty when facts are incomplete.
- Use first-person sparingly and intentionally.
- Prefer claims like "helped build", "led migration planning", or "owned deployment automation" only when those claims are supported by user-provided facts.
- Avoid claiming production scale, security impact, cost savings, reliability improvements, compliance outcomes, or business impact without evidence.
- When a metric is included, ensure it came from user-provided source material or an existing approved resume/project note.

## Confidentiality

Do not publish:

- Customer names, private employer details, unreleased projects, internal code names, or private architecture.
- IP addresses, hostnames, account IDs, tenant IDs, repository names, cloud resource names, cluster names, secrets, tokens, keys, or internal URLs.
- Incident details that could identify an organization or expose weaknesses.
- Screenshots containing private dashboards, tickets, alerts, repositories, calendars, email, logs, or chat.
- Internal diagrams, routing details, firewall rules, naming conventions, or access models that are not explicitly approved for public use.
- Contract details, statements of work, pricing, legal terms, or private client communications.

Sanitize case studies so they describe the engineering problem, constraints, approach, and outcome without exposing sensitive implementation details.

## Public-Sector And Employer-Safe Wording

- Keep descriptions of government, employer, and client work factual and sanitized.
- Avoid language that implies speaking on behalf of an employer, agency, client, or vendor.
- Avoid details that could be interpreted as disclosing internal security posture, operational weaknesses, procurement details, or incident response specifics.
- Prefer generalized descriptions such as "large enterprise Azure environment", "hybrid cloud network", or "public-sector environment" when specific naming is not necessary.

## Professional Tone

Use language that is:

- Specific.
- Calm.
- Evidence-driven.
- Technically literate.
- Understandable to engineering leaders and senior peers.

Avoid:

- Hype-heavy phrases like "world-class", "rockstar", "10x", or "guru".
- Dense buzzword strings.
- Generic AI-sounding phrases like "leveraging cutting-edge technology", "passionate about innovation", or "transforming businesses through technology".
- AI/process/scaffold language such as "template selection", "next step", "generated by AI", "prompt", "agent", or "placeholder" in public-facing pages.
- Overly casual jokes in core professional pages.
- Negative commentary about employers, teams, products, or vendors.

## Cloud And Platform Specificity

When relevant and supported by source material, name concrete domains:

- Infrastructure as code.
- CI/CD and release automation.
- Containers and orchestration.
- Cloud networking and identity.
- Observability, logging, metrics, tracing, and alerting.
- Reliability engineering and incident response.
- Security controls, least privilege, secrets handling, and compliance support.
- Cost management and operational efficiency.
- Developer experience and internal platforms.

Tie tools to outcomes. A list of technologies is less persuasive than a short description of how they were used.

Do not add technologies such as Kubernetes, containers, distributed systems, machine learning, or production SRE practices unless they are supported by user-provided facts or existing approved content.

## Portfolio Case Study Pattern

When writing project summaries, prefer this structure:

1. Problem or context.
2. Constraints.
3. Approach.
4. Tools used.
5. Outcome, only if supported.
6. Public-safe note if details are intentionally generalized.

Keep case studies concise. Avoid turning sanitized summaries into internal implementation documents.

## Accessibility And Readability

- Use semantic headings in order.
- Keep paragraphs short.
- Use descriptive link text instead of "click here".
- Provide alt text for meaningful images.
- Avoid text embedded in images unless the same information appears in HTML text.
- Keep acronyms expanded on first use when the audience may be mixed.

## SEO And Metadata

- Keep `_config.yml` metadata accurate and professional.
- Use page titles that describe the actual page.
- Avoid keyword stuffing.
- Ensure canonical URLs and base URLs remain compatible with GitHub Pages.

## External Links

- Link only to trustworthy, professional destinations.
- Prefer official documentation, personal profiles, published articles, and public repositories.
- Check links for accidental tracking parameters or private tokens.
- Use HTTPS links.
