# Security Rules

This site is public, static, and deployed through GitHub Pages. Security review is required for any change that affects dependencies, build configuration, external content, forms, scripts, workflows, or published technical details.

## Secret Handling

- Never add secrets, credentials, API keys, tokens, private keys, session cookies, signed URLs, `.env` files, or password material.
- Do not include realistic placeholder secrets. Use obviously fake examples like `example-token`.
- Before publishing logs, config, screenshots, or snippets, inspect for account IDs, hostnames, internal URLs, email addresses, usernames, and resource names.
- If secret exposure is suspected, stop and notify the user. Do not attempt rotation unless explicitly asked and properly scoped.

## Dependency Safety

- Keep gems minimal and compatible with the current Jekyll setup.
- Prefer GitHub Pages-supported plugins and conservative versions.
- Do not add npm, JavaScript bundlers, or front-end frameworks unless there is a clear need and user approval.
- Review new dependencies for maintenance quality, transitive risk, licensing, and deployment compatibility.
- Do not add remote install scripts or curl-pipe-shell instructions.

## Third-Party Content

Require explicit user approval before adding:

- Analytics.
- Tag managers.
- Comment systems.
- Newsletter forms.
- Contact forms.
- Calendly or scheduling embeds.
- Social widgets.
- Remote JavaScript.
- Iframes.
- CDN-hosted assets.

If approved, prefer privacy-preserving, minimal, documented integrations.

## Browser And Content Security

- Avoid inline scripts and unsafe HTML.
- Prefer plain Markdown and static HTML generated by Jekyll.
- If custom HTML is needed, keep it simple and inspect for XSS risk.
- Do not render untrusted user input.
- Use `rel="noopener noreferrer"` for external links opened in new tabs.
- Do not add mixed-content HTTP resources.

## GitHub Pages And CI/CD

- Do not alter deployment workflows without explicit instruction.
- Keep GitHub Actions permissions least-privilege if workflows are added later.
- Pin action versions to immutable SHAs or trusted major versions according to the user's preference.
- Do not expose secrets to pull requests from forks.
- Avoid workflows that publish artifacts containing generated files with sensitive local paths or environment data.

## Public Technical Disclosure

Portfolio case studies should avoid details that could help attack a real environment:

- No exact network topology.
- No exploitable misconfiguration details.
- No internal service names.
- No access patterns that reveal privileged paths.
- No incident timelines tied to identifiable organizations.
- No screenshots of dashboards, cloud consoles, alerts, tickets, or repositories unless fully sanitized.

Discuss security work at the control, process, and outcome level.

## Review Triggers

Perform an explicit security pass when changes include:

- `_config.yml`, `Gemfile`, GitHub workflows, layouts, includes, scripts, or HTML.
- External links, images, downloads, embeds, or forms.
- Project case studies involving production infrastructure.
- Resume-like content copied from private documents.
- Generated assets or screenshots.
